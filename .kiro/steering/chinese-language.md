# 一、语言总则

- 与用户的所有对话，必须始终使用中文
- 所有解释、分析、提问、总结，必须使用中文
- 禁止在输出中出现英文完整句子（除非符合“允许英文范围”）


# 二、允许使用英文的范围

仅允许在以下场景中使用英文内容：

- 编程语言关键字（if / for / return 等）
- 标准技术名词与协议字段：
  - HTTP / MQTT / BLE / JSON / API / SDK / CI / CD
- 第三方库、框架、工具名称
- 标准错误码、协议字段名、枚举值

所有英文内容，**如涉及业务或逻辑含义，必须配套中文说明**

# 三、中文工程表达规范
- 使用中国软件工程师的日常工程表达方式
- 避免以下风格：
  - 翻译腔
  - 学术论文腔
  - 营销文案腔
- 追求目标：
  - 简洁
  - 明确
  - 可执行
  - 可落地

推荐示例：
> “该模块用于处理设备状态上报逻辑”

不推荐示例：
> “该模块旨在实现对设备状态变化的综合管理能力”


# 四、命名语言规则
- 代码中的以下内容必须使用英文：
  - 变量名
  - 函数名
  - 类名
  - 接口名
- 禁止使用拼音命名（如：dengZhuCount、yongHuList）
- 中文仅允许用于：
  - 代码注释
  - 文档说明
  - 日志与错误提示

# 五、代码注释规则
- 所有代码注释必须使用中文
- 必须包含：
  - 模块说明
  - 关键逻辑说明
  - 复杂条件解释
- 注释目标：
  - 让未参与开发的工程师可以快速理解

# 六、生成使用中文
 - requirements.md 生成使用中文
 - design.md 生成使用中文
 - tasks.md 生成使用中文

# 七、异常、日志与提示规范

- 所有异常信息必须使用中文
- 所有日志内容必须使用中文
- 用户可见提示严禁英文
- 技术错误需采用以下格式：

示例：
```ts
throw new Error("设备未连接（device_not_connected）");
```

# 八、Git 提交信息规范

- commit message 使用中文
- 格式：`<类型>: <简要描述>`
- 类型包括：功能、修复、重构、文档、测试、样式、构建

示例：
```
功能: 新增设备状态上报接口
修复: 解决蓝牙连接超时问题
重构: 优化用户认证模块结构
文档: 更新 API 接口说明
测试: 补充设备连接单元测试
```
